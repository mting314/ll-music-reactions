# Data pipeline (GCP)

Replaces the hardcoded `src/data/*.json` imports with a **Cloud SQL (Postgres)**
database that is **refreshed daily** using the same scraper scripts as
[`hamzaabamboo/ll-sorter-scripts`](https://github.com/hamzaabamboo/ll-sorter-scripts).

## Architecture

```
Cloud Scheduler (daily 03:00 UTC)
        │  POST :run
        ▼
Cloud Run Job  ── clone upstream scrapers (pinned) + seed baseline
 (ll-data-refresh) ── run update.ts (DATA_ONLY) + parse-discography
        │             ── load canonical JSON → Postgres
        ▼
Cloud SQL (Postgres, instance "ll-data", db "llmusic")
        ▲
        │  SELECT
Cloud Run Service (ll-data-api)  ──  GET /data  ──►  Frontend (DataProvider)
```

The frontend fetches `GET /data` at load time when `VITE_DATA_API` is set;
otherwise it falls back to the bundled `src/data` snapshot, so the app always
works (and the live site is unaffected until the API is provisioned).

## Pieces

| Path | What |
|------|------|
| `schema.sql` | Postgres tables (document-style `id` + `JSONB`, plus a `builds` log) |
| `db.ts` | Shared Bun SQL connection (`DATABASE_URL`) |
| `apply-schema.ts` | Apply the schema (idempotent) |
| `load-to-db.ts` | Upsert canonical JSON → Postgres (also used to seed from `src/data`) |
| `run-refresh.ts` | Daily job: clone+run scrapers, then load to DB |
| `Dockerfile` | Cloud Run Job image (Bun + git) |
| `../data-api/server.ts` | Cloud Run service exposing `GET /data` |
| `setup-gcp.sh` | One-time provisioning (Cloud SQL, services, scheduler) |

## How the refresh reuses the upstream "system"

`run-refresh.ts` clones the upstream scraper repo at a pinned commit
(`SCRAPER_REF`) and a baseline data repo (`SEED_REPO`, default
`hamproductions/the-sorter`) for the `data/` + `data/raw/` the *incremental*
scrapers diff against. It then runs the upstream `update.ts` (`DATA_ONLY=1`) plus
`parse-discography`, producing the canonical JSON, which `load-to-db.ts` upserts.

`series-info.json` / `series.json` are not regenerated by the scrapers; they are
carried forward from the seed clone.

## Setup (run once, after review)

```bash
# Cloud SQL is billable + always-on, so this is intentionally manual.
export DB_PASSWORD='<a strong password>'
./pipeline/setup-gcp.sh

# Seed the DB from the committed snapshot (or just run the job once):
gcloud run jobs execute ll-data-refresh --region us-central1

# Point the frontend at the API and redeploy Pages:
#   .env.production -> VITE_DATA_API=https://ll-data-api-....run.app
```

## Local development

```bash
cd pipeline && bun install
export DATABASE_URL='postgres://user:pass@localhost:5432/llmusic'
bun run schema      # create tables
bun run seed        # load src/data snapshot into the DB
bun run ../data-api/server.ts   # serve /data locally
```

## ⚠️ Validation note

The DB schema, ingestion, read API, and frontend wiring are environment-
independent and verified by CI. The **scrape step** hits external sites and
depends on the upstream repos' layout, so it can only be validated in the live
Cloud Run Job — run it once and check the `builds` table after provisioning.
