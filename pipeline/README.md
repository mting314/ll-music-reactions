# Data pipeline (GCP)

Replaces the hardcoded `src/data/*.json` imports with a dataset stored in
**Google Cloud Storage** that is **refreshed daily** using the same scraper
scripts as
[`hamzaabamboo/ll-sorter-scripts`](https://github.com/hamzaabamboo/ll-sorter-scripts).

GCS (object storage) is used instead of a database because the app loads the
whole dataset and filters client-side — there are no server-side queries to run.
This keeps the cost at roughly **$0/month** (vs. ~$10/mo for an always-on
Cloud SQL instance) and removes the need for a read API service entirely.

## Architecture

```
Cloud Scheduler (daily 03:00 UTC)
        │  POST :run
        ▼
Cloud Run Job  ── clone upstream scrapers (pinned) + seed baseline
 (ll-data-refresh) ── run update.ts (DATA_ONLY) + parse-discography
        │             ── assemble dataset.json
        ▼
GCS bucket  gs://<project>-ll-data/dataset.json  (public, CDN-cached)
        ▲
        │  fetch (direct)
Frontend (DataProvider, VITE_DATA_URL)
```

The frontend fetches the dataset object directly when `VITE_DATA_URL` is set;
otherwise it falls back to the bundled `src/data` snapshot, so the app always
works (and the live site is unaffected until provisioned).

## Pieces

| Path | What |
|------|------|
| `build-dataset.ts` | Assemble canonical JSON → one `dataset.json` (local file or GCS) |
| `gcs.ts` | Upload to GCS via the metadata-server token (no SDK) |
| `run-refresh.ts` | Daily job: clone+run scrapers, build dataset, upload to GCS |
| `Dockerfile` | Cloud Run Job image (Bun + git) |
| `setup-gcp.sh` | One-time provisioning (bucket, job, scheduler) |

## How the refresh reuses the upstream "system"

`run-refresh.ts` clones the upstream scraper repo at a pinned commit
(`SCRAPER_REF`) plus a baseline data repo (`SEED_REPO`, default
`hamproductions/the-sorter`) for the `data/` + `data/raw/` the *incremental*
scrapers diff against. It runs the upstream `update.ts` (`DATA_ONLY=1`) plus
`parse-discography`, then assembles + uploads `dataset.json`.

`series-info.json` / `series.json` aren't regenerated by the scrapers; they are
carried forward from the seed clone.

## Setup (run once, after review)

```bash
./pipeline/setup-gcp.sh            # bucket + job + scheduler, runs job once
# then point the frontend at the dataset and redeploy Pages:
#   .env.production -> VITE_DATA_URL=https://storage.googleapis.com/<bucket>/dataset.json
```

## Local development

```bash
cd pipeline && bun install
bun run build:local        # assemble src/data -> ./dataset.json (no network)
# serve it and point the app at it:
#   VITE_DATA_URL=http://localhost:8000/dataset.json bun dev
```

## ⚠️ Validation note

`build-dataset.ts` (assembly), `gcs.ts`, and the frontend wiring are environment-
independent and verified by CI / local runs. The **scrape step** in
`run-refresh.ts` hits external sites and depends on the upstream repos' layout,
so it can only be validated in the live Cloud Run Job — run it once and check the
uploaded `dataset.json`.
